# ExpoCore Investor Platform — Essential Use Cases (Detailed)

---

## Essential Use Cases Table

| UC ID | Use Case | Detailed Step-by-Step Description | Req # |
|-------|----------|-----------------------------------|-------|
| **UC02** | **Login** | **1.** The investor opens the app; the splash screen checks SharedPreferences for an existing token.<br>**2.** If no token is found, the app navigates to the Login screen.<br>**3.** The investor enters their email address and password.<br>**4.** The investor optionally toggles "Remember Me".<br>**5.** The investor taps "Login"; the controller validates the form fields (non-empty, valid email format).<br>**6.** If validation fails, inline error messages are shown on the respective fields; no API call is made.<br>**7.** If validation passes, `StatusRequest.loading` is set and a loading indicator appears on the button.<br>**8.** `LoginController` calls `POST /auth/login` with `{email, password, remember_me}`.<br>**9a.** On HTTP 200: the JWT token and company name are extracted from the response and saved to SharedPreferences. The app navigates to `/dashboard`.<br>**9b.** On any error: `StatusRequest.failure` is set, loading disappears, and a red snackbar displays the API error message or a default message. | FR02, FR05 |
| **UC01** | **Register Account** | **1.** The investor navigates to the Register screen from the Login screen.<br>**2.** The investor fills in: Company Name, Trade Name, Email, Phone, Website, Activity Type (dropdown), Password, and Confirm Password.<br>**3.** The investor checks the "I agree to Terms & Conditions" checkbox.<br>**4.** The investor taps "Register".<br>**5.** The controller validates all fields: required fields non-empty, email format valid, password ≥ 8 characters, passwords match.<br>**6.** If validation fails, inline errors are shown on each invalid field.<br>**7.** If the Terms checkbox is not checked, a snackbar error is displayed and no API call is made.<br>**8.** On valid form, `StatusRequest.loading` is set.<br>**9.** `RegisterController` calls `POST /auth/register` with all form fields.<br>**10a.** On success: the investor is redirected to the Login screen with a success snackbar "Account created successfully".<br>**10b.** On error: `StatusRequest.failure` is set and the API error message is shown in a snackbar. | FR01 |
| **UC11** | **Book a Booth** | **1.** The investor reaches the Booking Request screen either via the 3D Map ("Proceed to Booking" button after selecting a booth) or via a Booth Detail screen.<br>**2.** The screen pre-populates the chosen booth's details: number, exhibition, area, price per day, location, and amenities.<br>**3.** The investor selects the booking duration in days using a stepper control.<br>**4.** The investor optionally toggles any of four add-on services: Screen Display (SAR 500), Setup Service (SAR 800), Security Service (SAR 300), Cleaning Service (SAR 200).<br>**5.** The total cost updates reactively as duration or services change.<br>**6.** The investor enters optional notes in the text field.<br>**7.** The investor taps "Submit Booking Request"; `isSubmitting` is set to `true` and the button shows a spinner.<br>**8.** `BookingController` calls `POST /booths/book` with `{booth_id, duration_days, notes, screen_service, setup_service, security_service, cleaning_service, total_price}`.<br>**9a.** On success: the investor is returned to the previous screen and a snackbar shows "Your request is under review ⏳".<br>**9b.** On failure: `isSubmitting` is reset, a red snackbar displays the API error. | FR19, FR20, FR21 |
| **UC15** | **Create Event** | **1.** The investor navigates to Create Event from the Events screen tab.<br>**2.** The investor selects an Exhibition from the dropdown (populated from their booked booths).<br>**3.** The investor selects a Booth within the chosen exhibition (filtered list).<br>**4.** The investor enters: Event Name, Event Type (one of 8 types from dropdown), Date (date picker), Start Time (time picker), Duration (days stepper), Max Participants, and Description.<br>**5.** Optionally, the investor enables "Bookable Seats", which reveals Seat Count and Ticket Price fields.<br>**6.** The investor chooses whether the event uses a General Invitation or ticket-based entry.<br>**7.** Optionally, the investor adds a Video Promo URL.<br>**8.** The investor taps "Publish Event"; the controller validates: all required fields present, type selected, exhibition selected, booth selected.<br>**9.** If any validation fails, a warning snackbar indicates the specific missing field.<br>**10.** On valid form, `isCreating` is set to `true` and the button shows a loader.<br>**11.** `EventsController` calls `POST /investor/events` with the full event payload.<br>**12a.** On success: the events list is refreshed from the API, the create form resets, the sheet closes, and a green snackbar confirms the event was published.<br>**12b.** On failure: `isCreating` resets and an error snackbar is shown. | FR27–FR31 |
| **UC20** | **Manage Ticket Requests** | **1.** The investor opens one of their events from the My Events tab.<br>**2.** The investor taps "View Ticket Requests"; `EventsController.getTicketRequests(eventId)` is called, which triggers `GET /investor/events/{id}/ticket-requests` if the data is not yet loaded.<br>**3.** The screen displays a list of requests, each showing: requester name, phone, email, request date, and current status badge (Pending / Approved / Rejected).<br>**4.** For a Pending request, the investor taps "Approve".<br>**5.** The controller calls `PATCH /investor/events/{eventId}/ticket-requests/{reqId}` with `{action: 'approve'}`.<br>**6.** On success: the request's status is updated to "Approved"; a ticket number (ECT-XXX) and QR code data are assigned; the list refreshes; a green snackbar confirms QR generation.<br>**7.** For a Pending request, the investor taps "Reject".<br>**8.** The controller calls `PATCH` with `{action: 'reject'}`.<br>**9.** The request's status updates to "Rejected"; a red snackbar confirms the rejection. | FR35–FR37 |
| **UC18** | **Book Sponsorship** | **1.** The investor opens the "Exhibition Events" tab within the Events screen; only events belonging to exhibitions where they have a booth are shown.<br>**2.** The investor taps on a sponsor event to view its detail: name, type, date, venue, description, and available duration options (each with label, days, and price).<br>**3.** The investor selects one Duration Option from the list.<br>**4.** The investor reviews or edits their company details: Company Name, Website, Phone, and Products/Services.<br>**5.** The investor taps "Book Sponsorship Slot"; `isBooking` is set to `true`.<br>**6.** `EventsController` calls `POST /investor/sponsorships` with `{event_id, selected_duration_label, selected_days, price, company_name, company_website, company_phone, product_names}`.<br>**7a.** On success: the sponsorships list is refreshed from the API; the booking sheet closes; a green snackbar confirms the request was submitted.<br>**7b.** On failure: a local `SponsorshipBookingModel` is added optimistically; the sheet closes with confirmation. | FR32–FR34 |
| **UC28** | **Download Report** | **1.** The investor is on the Reports screen viewing the filtered list.<br>**2.** The investor taps a specific report card to open the Report Detail screen.<br>**3.** The investor taps the "Download" button and selects a format: PDF or CSV.<br>**4.** `isDownloading` is set to `true`; a linear progress indicator appears.<br>**5.** `ReportsController` calls `GET /investor/reports/{id}/download?format={pdf|csv}`.<br>**6.** A simulated progress animation increments from 0% to 100% over ~1.5 seconds (to accommodate network latency).<br>**7.** On completion, `isDownloading` resets to `false`.<br>**8.** A snackbar confirms "Report downloaded in {FORMAT} format". | FR47, FR48 |
| **UC30** | **Chat with Exhibition Department** | **1.** The investor opens the Messages screen; `MessagesController` calls `GET /investor/messages` to load all conversations.<br>**2.** Each conversation card shows the exhibition initials avatar (colour-coded), exhibition name, last message preview, and unread count badge.<br>**3.** The investor taps a conversation card; `openConversation(id)` is called, resetting the unread count to 0.<br>**4.** The controller calls `GET /investor/messages/{id}` to load the full message thread.<br>**5.** The Chat screen displays messages in a scrollable list: investor messages (right, primary colour) and department messages (left, surface colour).<br>**6.** The investor types a message in the input field and taps Send.<br>**7.** The message is added to the list immediately (optimistic update) and the input is cleared.<br>**8.** The controller calls `POST /investor/messages/{convId}/send` with `{text}`.<br>**9a.** On API success: no further action needed; the server reply arrives via polling or push (future).<br>**9b.** On API failure: a simulated reply "Thank you for contacting us. Support will respond shortly." is shown after 1.2 seconds. | FR49–FR51 |
| **UC14** | **Manage Booth Profile** | **1.** The investor navigates to one of their booths from the My Booths screen and taps "Manage Profile".<br>**2.** The `BoothManagementController` calls `GET /investor/booths/{id}/profile` to load existing profile data.<br>**3.** The screen displays editable sections: Company Nature, Services & Products, Headquarters, Social Links, Product Images, and Booth Images.<br>**4.** The investor edits the Company Nature, Services, or Headquarters text fields.<br>**5.** The investor adds a social link by typing a URL and tapping Add; it appears in the links list. Links can be removed individually.<br>**6.** The investor taps "Add Product Image" or "Add Booth Image" to append a new image entry.<br>**7.** The investor taps "Save Profile".<br>**8.** `isSaving` is set to `true`; a loading indicator replaces the save button.<br>**9.** The controller calls `PUT /investor/booths/{id}/profile` with the complete profile payload (all text fields + arrays of links/images).<br>**10a.** On success: `isSaving` resets; a green snackbar confirms "Company information saved successfully".<br>**10b.** On failure: `isSaving` resets; a red snackbar shows the API error. | FR22–FR26 |
| **UC41** | **Edit Company Profile** | **1.** The investor navigates to the Profile screen; `ProfileCompanyController` calls `GET /investor/profile` on `onInit`.<br>**2.** The loaded data populates the fields: Company Name, Email, Phone, Website, and Bio in read-only display mode.<br>**3.** The investor taps the "Edit" icon to enter edit mode; all fields become editable text inputs.<br>**4.** The investor modifies any of the fields.<br>**5.** The investor taps "Save Changes"; `isSaving` is set to `true`.<br>**6.** `ProfileCompanyController` calls `PUT /investor/profile` with `{company_name, email, phone, website, bio}`.<br>**7a.** On success: the company name is also updated in SharedPreferences; `isEditing` is set to `false`; a green snackbar confirms save.<br>**7b.** On failure: `isSaving` resets; a red snackbar shows the API error. | FR61, FR62 |
| **UC38** | **Mark Notification Read** | **1.** The investor opens the Notifications screen; `NotificationsController` calls `GET /investor/notifications` on `onInit`.<br>**2.** Notifications are displayed in a list; unread items are visually highlighted with a coloured indicator.<br>**3.** The investor taps on an unread notification.<br>**4.** The controller immediately replaces the item in the list with a copy where `isRead = true` (optimistic update); the list refreshes reactively.<br>**5.** The controller calls `PATCH /investor/notifications/{id}/read` in the background.<br>**6.** The unread count badge in the app bar updates immediately. | FR58, FR59 |
| **UC04** | **Logout** | **1.** The investor navigates to the Settings screen.<br>**2.** The investor taps "Logout".<br>**3.** A confirmation dialog appears: "Are you sure you want to log out?"<br>**4.** The investor taps "Confirm".<br>**5.** `isLoggingOut` is set to `true`; the button shows a loading indicator.<br>**6.** `SettingsController` calls `POST /auth/logout` to invalidate the server-side token.<br>**7.** Regardless of API result, `Services.clearSession()` is called, which removes the JWT token and company name from SharedPreferences.<br>**8.** `isLoggingOut` resets; the app navigates to the Login screen, clearing the navigation stack. | FR04, FR05 |
